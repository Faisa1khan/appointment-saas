## Overview

### URL Structure

```
/api/v1/auth/*          — Authentication (no org scope)
/api/v1/public/{slug}/* — Public booking API (no auth required)
/api/v1/owner/*         — Owner & Staff API (auth required, org from JWT)
```

- **Public endpoints** use the business `slug` to identify the organization.
- **Authenticated endpoints** derive the organization from the user's JWT session. No `organizationId` in the URL.
- **Future:** `/api/v1/admin/*` for platform-level administration.

### Authentication

- Supabase Auth JWT token, passed as `Authorization: Bearer <token>`.
- Server-side Route Handlers validate tokens using the Supabase server client.
- The org is resolved from `organization_members` using `auth.uid()`.
- Public endpoints require no token.

### Request Format

- All request bodies: JSON (`Content-Type: application/json`)
- Dates: `YYYY-MM-DD`
- Times: `HH:MM` (24-hour)
- Monetary values: integers in smallest currency unit (e.g. `500` = 5.00)
- UUIDs: standard UUID v4

### Response Format

**Success:**
```json
{ "data": { ... } }
```

**List (paginated):**
```json
{
  "data": [ ... ],
  "meta": {
    "total": 42,
    "page": 1,
    "limit": 20,
    "sort": "created_at",
    "order": "desc"
  }
}
```

**Error:**
```json
{
  "error": {
    "code": "BOOKING_CONFLICT",
    "message": "The selected time slot is no longer available."
  }
}
```

### Pagination Standard

All list endpoints use these query params:

| Param | Type | Default | Notes |
|---|---|---|---|
| `page` | integer | `1` | Page number |
| `limit` | integer | `20` | Results per page, max `100` |
| `sort` | string | `created_at` | Column to sort by |
| `order` | string | `desc` | `asc` or `desc` |

### Standard Error Codes

| HTTP | Code | Description |
|---|---|---|
| 400 | `VALIDATION_ERROR` | Request body failed Zod validation |
| 401 | `UNAUTHORIZED` | Missing or invalid auth token |
| 403 | `FORBIDDEN` | Authenticated but not allowed |
| 404 | `NOT_FOUND` | Resource not found |
| 409 | `BOOKING_CONFLICT` | Time slot conflict on booking creation |
| 409 | `INVALID_STATUS_TRANSITION` | Booking status cannot transition this way |
| 422 | `BUSINESS_CLOSED` | Organization closed on requested date |
| 422 | `RESOURCE_UNAVAILABLE` | Resource has no availability |
| 422 | `ADVANCE_BOOKING_TOO_SOON` | Within org's `min_advance_minutes` window |
| 422 | `CANCELLATION_WINDOW_CLOSED` | Within org's `cancellation_cutoff_hours` window |
| 429 | `RATE_LIMITED` | Too many requests |
| 500 | `INTERNAL_ERROR` | Unexpected server error |

### Booking Reference Format

Every booking has a human-friendly reference customers can use when contacting the business.

```
Format:  BK-{YYYYMMDD}-{NNNNNN}
Example: BK-20260720-000042
```

- Generated by the application at booking creation time.
- Unique within an organization.
- Always returned alongside the booking `id`.

### Idempotency

> **Future:** Booking creation will support an `Idempotency-Key` header to safely retry requests without creating duplicate bookings. Not implemented in V1.

### GET `/api/v1/public/{slug}/availability`

Fetch available booking slots for a given date and service selection.

**Query params:**

| Param | Type | Required | Notes |
|---|---|---|---|
| `date` | YYYY-MM-DD | Yes | |
| `service_ids` | comma-separated UUIDs | Yes | |
| `resource_id` | UUID | No | Filter by specific resource |

**Business logic:**
1. Check if date is a business closure → `BUSINESS_CLOSED`.
2. Determine business hours or resource schedule for the date.
3. Calculate total service duration from `service_ids`.
4. Generate slots at `booking_interval` from open to (close − total duration).
5. Exclude slots starting within `min_advance_minutes` from now.
6. Exclude slots where resource has overlapping `CONFIRMED`/`CHECKED_IN` bookings.
7. Return slots.

**Response `200`:**
```json
{
  "data": {
    "date": "2026-07-25",
    "booking_interval": 30,
    "total_duration": 60,
    "slots": [
      { "start_time": "09:00", "end_time": "10:00", "available": true },
      { "start_time": "09:30", "end_time": "10:30", "available": false }
    ]
  }
}
```

**Errors:** `VALIDATION_ERROR`, `NOT_FOUND`, `BUSINESS_CLOSED`, `RESOURCE_UNAVAILABLE`

---

### POST `/api/v1/public/{slug}/customers`

Create or look up a customer. Used during the guest booking flow.

**Request:**
```json
{ "name": "Fatima Khan", "phone": "+92 301 9876543", "email": "fatima@example.com" }
```

**Validation:**
- `name`: required
- At least one of `phone` or `email` required
- If a customer with the same phone already exists in this org → return that customer (no duplicate)

**Response `201`:** Customer object with `id`.

---

### POST `/api/v1/public/{slug}/bookings`

Create a booking. Public, no auth required.

**Request:**
```json
{
  "customer_id": "uuid",
  "service_ids": ["uuid", "uuid"],
  "date": "2026-07-25",
  "start_time": "10:00"
}
```

**Validation:**
- `customer_id`: required, must belong to this org
- `service_ids`: required, non-empty, all active in this org
- `date`: today or future
- `start_time`: required
- `start_time` must not be within `min_advance_minutes` from now

**Business logic:**
1. Calculate `end_time = start_time + total service duration`.
2. Check org is not closed on date.
3. Auto-assign first available resource (if any active resources exist).
4. Conflict check on assigned resource.
5. Create booking with `source = ONLINE`, `status = CONFIRMED`.
6. Generate `reference` (e.g. `BK-20260720-000042`).
7. Create `booking_services` snapshot.
8. Send WhatsApp confirmation (fire-and-forget).

**Response `201`:**
```json
{
  "data": {
    "id": "uuid",
    "reference": "BK-20260720-000042",
    "date": "2026-07-25",
    "start_time": "10:00",
    "end_time": "11:00",
    "status": "CONFIRMED",
    "source": "ONLINE",
    "services": [
      { "service_name": "Haircut", "duration": 30, "price": 500, "quantity": 1, "total": 500 }
    ],
    "resource": { "name": "Barber 1" },
    "business": { "name": "Amir Barber Shop", "phone": "+92 300 1234567" }
  }
}
```

**Errors:** `VALIDATION_ERROR`, `NOT_FOUND`, `BOOKING_CONFLICT`, `BUSINESS_CLOSED`, `ADVANCE_BOOKING_TOO_SOON`

---

### GET `/api/v1/public/{slug}/bookings/lookup`

Retrieve a guest customer's active bookings by phone number.

**Auth:** No (rate-limited: 10 requests per phone per 15 min)

**Query params:**

| Param | Required | Notes |
|---|---|---|
| `phone` | Yes | Customer phone number |

**Business logic:**
- Find customer in org by phone.
- Return `CONFIRMED` and `CHECKED_IN` bookings only.
- If no match → return empty list (never 404 — avoids leaking phone existence).

**Response `200`:**
```json
{
  "data": {
    "customer": { "name": "Fatima Khan", "phone": "+92 301 9876543" },
    "bookings": [
      {
        "id": "uuid",
        "reference": "BK-20260720-000042",
        "date": "2026-07-25",
        "start_time": "10:00",
        "end_time": "11:00",
        "status": "CONFIRMED",
        "services": [{ "service_name": "Haircut", "duration": 30 }],
        "resource": { "name": "Barber 1" }
      }
    ]
  }
}
```

**Errors:** `VALIDATION_ERROR`, `RATE_LIMITED`

---

### GET `/api/v1/public/{slug}/bookings/{bookingId}`

Get a single booking. Customer can view their own booking via reference link.

**Response `200`:** Full booking object.

**Errors:** `NOT_FOUND`

---

### POST `/api/v1/public/{slug}/bookings/{bookingId}/cancel`

Customer cancels their own booking.

**Request:**
```json
{ "phone": "+92 301 9876543" }
```

Phone is used to verify the customer owns the booking (no auth token required).

**Business rules:**
- Only `CONFIRMED` bookings can be cancelled.
- Rejected if within `cancellation_cutoff_hours` → `CANCELLATION_WINDOW_CLOSED`.

**Response `200`:** Updated booking with `status = CANCELLED`.

**Errors:** `NOT_FOUND`, `INVALID_STATUS_TRANSITION`, `CANCELLATION_WINDOW_CLOSED`

---

### POST `/api/v1/public/{slug}/bookings/{bookingId}/reschedule`

Customer reschedules their own booking.

**Request:**
```json
{ "phone": "+92 301 9876543", "date": "2026-07-28", "start_time": "14:00" }
```

**Business rules:**
- Only `CONFIRMED` bookings.
- Backend re-validates availability for new date/time.
- Same services kept; new `end_time` calculated.

**Response `200`:** Updated booking.

**Errors:** `NOT_FOUND`, `BOOKING_CONFLICT`, `INVALID_STATUS_TRANSITION`, `ADVANCE_BOOKING_TOO_SOON`

### POST `/api/v1/owner/organization`

Create an organization. The authenticated user becomes OWNER.

**Auth:** Yes

**Request:**
```json
{
  "name": "Amir Barber Shop",
  "slug": "amir-barber",
  "phone": "+92 300 1234567",
  "email": "contact@amirbarber.com",
  "address": "Plot 5, DHA Phase 1, Karachi",
  "timezone": "Asia/Karachi",
  "booking_interval": 30,
  "min_advance_minutes": 60,
  "cancellation_cutoff_hours": 2
}
```

**Validation:**
- `name`: required, min 2 chars
- `slug`: required, unique, lowercase alphanumeric + hyphens, min 3 chars
- `timezone`: valid IANA string
- `booking_interval`: one of `[15, 30, 60]`
- `phone` or `email`: at least one required
- `min_advance_minutes`: non-negative integer, default `0`
- `cancellation_cutoff_hours`: non-negative integer, default `0`

**Response `201`:** Organization object.

**Errors:** `VALIDATION_ERROR`, `UNAUTHORIZED`

---

## Status Transition Summary

| Current | Allowed Transitions | Who |
|---|---|---|
| `CONFIRMED` | `CHECKED_IN` | OWNER, STAFF |
| `CONFIRMED` | `CANCELLED` | OWNER, STAFF, Customer |
| `CONFIRMED` | `NO_SHOW` | OWNER, STAFF |
| `CHECKED_IN` | `COMPLETED` | OWNER, STAFF |
| `COMPLETED` | — | No transitions |
| `CANCELLED` | — | No transitions |
| `NO_SHOW` | — | No transitions |

Any transition not in this table is rejected with `INVALID_STATUS_TRANSITION`.
